The Intake

An IT helpdesk triage agent that classifies, enriches, and routes — and only auto-resolves the two cases it has earned.

Team: Intake One · Built on the Claude Agent SDK (TypeScript) · 200 requests/day · target time-to-first-response under 5 min on routine, under 15 min on escalations.

02 · Problem

The problem we picked

Today

  • ~200 inbound requests/day
  • 3 channels — email, web form, chat
  • L1 rotation hand-triages everything
  • Median time-to-first-response: ~3 hours
  • Mistakes: misroutes, forgotten frozen accounts, password resets that should have escalated

Goal

  • ≤ 5 min on routine work
  • ≤ 15 min on escalations
  • 0 missed escalations on the cases that matter (frozen, P1, phishing, exec)
  • Auditable from the log alone
  • Defensible to Legal before launch
03 · Mandate

The agent's job — on one page

Decides alone

Only when ALL hold:

  • Auto-resolvable KB hit (KB-001 or KB-005)
  • Confidence ≥ 0.85
  • Impact = low
  • User not VIP, not frozen

Always escalates

  • Confidence < 0.85
  • VIP, frozen, exec mention
  • Phishing, P1, MFA
  • Unknown category (yes, valid output)
  • Medium/high dollar impact

Never touches

  • Frozen accounts (any write)
  • MFA reset / device replacement
  • Production P1 incidents
  • Phishing reports
  • Anything not in the catalog of categories

Default deny. Adding to auto-resolve requires (a) ≥ 50 labeled examples, (b) category owner sign-off, (c) ≥ 95% accuracy on the eval suite.

Coordinator + 3 specialists

Each specialist stays under the 5-tool reliability cliff. Subagents do not inherit context — every prompt is built explicitly by the coordinator.

05 · Core insight

The Resolver never sees the body

Why this matters

Prompt injection lives in attacker-controlled prose. If the write-capable subagent reads the inbound request, every clever instruction in the body is a potential foothold.

So we don't let it.

How

  • Classifier sees the body — it has to.
  • Enricher sees only the user identifier and the classification.
  • Resolver sees only the structured plan: {kind, queue, summary, kb_id}

What this buys

structural An attacker can hijack the classifier's reasoning, but the worst they can change is the label. Humans see and approve the actual write.

auditable The coordinator's plan is plain TypeScript output — auditable, replayable, diffable.

composable The brake (next slide) layers on top of this: hook + canUseTool catch what slipped through the structure.

06 · The brake

Two layers: hard stop + slow stop

PreToolUse hook (hard stop)

Deterministic. Runs before canUseTool. Cannot be argued with.

  • Frozen account → deny
  • PII patterns (SSN, credit card, IBAN) → deny
  • Exec mention on auto-resolve → deny
  • Prompt-injection patterns in tool input → deny
  • Phishing routed to wrong queue → deny
  • P1 routed to wrong queue → deny
  • KB not in auto-resolve whitelist → deny

canUseTool callback (slow stop)

Policy-as-code: category × confidence × impact × VIP.

if (vip)                            → escalate
if (confidence < 0.7)               → escalate
if (category in ALWAYS_ESCALATE)    → escalate
if (impact === "high")              → escalate
if (impact === "medium"
    && confidence < 0.9)            → escalate
if (confidence < 0.85)              → escalate
otherwise                           → allow

Both layers run on the Resolver only. The Classifier and Enricher are read-only; the SDK's allowed-tools list keeps them that way.

07 · Tools

6 tools, each ≤ 3 per specialist

ToolSpecialistWhat it does NOT do
search_kbClassifierfetch full body — use get_kb_article
get_kb_articleClassifiersearch by free text
lookup_userEnricherchange attributes; resolve display names
read_ticket_historyEnricherreturn ticket bodies (privacy)
auto_resolve ⚠Resolverhandle MFA, frozen, P1, phishing, non-whitelist KB
route_ticket ⚠Resolversend messages to users; close tickets

All errors return { isError: true, content: [{type:"text", text: JSON.stringify({reason_code, guidance})}] }. The agent can branch on the reason code and recover. Throwing would kill the loop.
